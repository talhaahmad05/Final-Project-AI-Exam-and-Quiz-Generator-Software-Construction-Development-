# PROCESS MODEL

For the development of the AI-Powered Quiz & Exam Platform, the Agile Scrum process model was selected as the foundational methodology. Agile was chosen over traditional sequential models, such as Waterfall or Spiral, primarily due to the highly dynamic nature of artificial intelligence integration. Since the project involved working with the Ollama local LLM (specifically the mistral:7b-instruct-q4_0 model), requirements and system behaviors were expected to evolve based on the model's actual performance. Waterfall's rigid, upfront design phase would not have accommodated the iterative tuning required for AI prompt engineering and response handling. Agile provided the necessary flexibility to adapt to these evolving technical challenges while ensuring continuous delivery of functional increments.

The project lifecycle was divided into five distinct sprints, each focusing on delivering specific, testable value. Sprint 1 was dedicated to the initial project setup, defining the SQLite database schema in `database/db_setup.py`, and establishing the role-based authentication module for Admin/Teacher and Student access. Sprint 2 expanded the core functionality by introducing the Question Bank and the Exam Builder, allowing educators to create MCQs, True/False, and Short Answer questions. Sprint 3 shifted focus to the student experience, implementing the live exam-taking interface with a countdown timer and the automatic grading systems for objective questions. Sprint 4 was the most critical phase, dedicated entirely to the integration of the six AI features via `ai_engine.py`, including the AI Question Generator, intelligent hints, and the streaming study chatbot. Finally, Sprint 5 encompassed comprehensive testing, code refactoring, and final deployment preparations.

Throughout the development cycle, the Scrum framework provided robust mechanisms for project management and team synchronization. The product backlog was continuously groomed to prioritize tasks based on their technical dependency and value to the end-user. Daily stand-up meetings, even when conducted as self-assessments, were instrumental in identifying blockers early, particularly during the complex AI integration phases where API response formats frequently shifted. These daily check-ins ensured that issues like UI freezing during AI calls were promptly identified and scheduled for resolution.

Furthermore, the sprint review process at the conclusion of each iteration significantly improved the final product's quality. By evaluating the working software increment, it became apparent when certain features, such as the initial streaming implementation for the MCQ generator, were prone to JSON parsing errors. The Agile methodology allowed the project to pivot quickly, replacing the streaming approach with blocking requests to stabilize the output. Ultimately, the iterative nature of Agile Scrum proved to be the ideal fit for this AI-integrated desktop application, ensuring that the final deliverable was both technically sound and aligned with the intended educational objectives.

# SOFTWARE PROCESS IMPROVEMENT (SPI)

Software Process Improvement (SPI) was a continuous and integral part of the AI-Powered Quiz & Exam Platform's development lifecycle. Following the principles of Agile Scrum, sprint retrospectives were conducted after each iteration to critically evaluate the development process and identify areas for enhancement. These retrospectives led to several targeted improvements that directly addressed performance bottlenecks and structural inefficiencies discovered during implementation. The primary goal of these SPI initiatives was to enhance the application's responsiveness, stability, and overall code maintainability.

The most critical process improvement (SPI 1) occurred during Sprint 3, following the discovery that the application's user interface was freezing during AI operations. To resolve this, `QThread` background workers were introduced into the architecture, ensuring that all intensive LLM calls within `ai_engine.py` were executed asynchronously, thereby maintaining a fluid user experience. Subsequent performance testing revealed that the AI model reloading between requests was causing unacceptable delays. Consequently, SPI 2 was implemented by adding a `keep_alive` parameter to the Ollama API payloads, ensuring the model remained active in memory. Additionally, SPI 3 involved reducing the `num_ctx` configuration from 4096 to 512 in `config.py` after memory profiling indicated that the larger context window was unnecessary for standard queries and was causing performance degradation.

Further process improvements focused on reliability and structural code quality. SPI 4 was initiated when the team encountered persistent JSON parse errors originating from the AI MCQ generator while using streaming requests. The process was adapted to switch from streaming to blocking requests specifically for generation tasks, which completely resolved the parsing instability. To further optimize the user experience, SPI 5 introduced a `warm_up_model()` function executed from `main.py` upon application startup, pre-loading the mistral model to eliminate the initial latency penalty.

Beyond AI-specific optimizations, architectural process improvements were also enacted. SPI 6 addressed code duplication by introducing the `FormValidator` utility class in the `utils/` directory, consolidating validation logic that was previously scattered across multiple UI modules. Finally, recognizing the complexity of the integrated system, SPI 7 established a policy of mandatory code reviews prior to merging any feature branch into the develop branch. This policy was instituted after several integration bugs surfaced in Sprint 2, and its implementation significantly reduced the defect rate in subsequent iterations. Each of these improvements was systematically measured through performance metrics and verified via automated testing to ensure they delivered tangible benefits to the software's quality.

# VERSION CONTROL

Version control was implemented rigorously throughout the development of the AI-Powered Quiz & Exam Platform to manage the evolving codebase and facilitate systematic feature integration. Git was selected as the distributed version control system, with GitHub serving as the remote repository host. This infrastructure provided a secure, centralized location for the project's assets while enabling robust tracking of all modifications. The use of Git was essential for maintaining the integrity of the application, particularly given the complex nature of integrating local AI models and coordinating multiple interconnected modules such as the UI, database DAOs, and the AI engine.

The project adopted the Git Flow branching strategy to organize development efforts logically and safely. The repository maintained a stable `main` branch, which exclusively held production-ready code, and a `develop` branch that served as the primary integration point for new features. Specific development tasks were isolated into dedicated feature branches, including `feature/auth`, `feature/question-bank`, `feature/exam-engine`, `feature/ai-integration`, and `feature/testing`. This isolation ensured that experimental AI implementations or complex UI modifications could be developed and tested independently without disrupting the functional core of the application residing in the `develop` branch.

To maintain a clear and informative project history, a strict commit message format was enforced. Every commit was required to follow the pattern "[Module] Action: Description", allowing for rapid comprehension of changes when reviewing the commit log. Before any feature branch could be merged into `develop`, a formal Pull Request (PR) process was mandatory. This process served as a gateway for peer review, ensuring that new code met the project's quality standards, adhered to PEP8 conventions, and included appropriate unit tests. Any merge conflicts that arose, particularly when multiple UI components required updates to the shared `config.py` or database schemas, were systematically resolved during the PR phase.

Furthermore, tag-based version releases were utilized to mark significant milestones in the software's evolution, progressing from early alpha builds (v0.1.0) through to the final production release (v1.0.0). This systematic approach to versioning not only provided a reliable mechanism for rolling back changes if critical bugs were discovered but also established a comprehensive audit trail of the project's development. The Git history detailed the precise evolution of features, such as the transition of database management into the DAO pattern, demonstrating how version control was indispensable for managing the project's complexity and ensuring a stable, high-quality final product.

# SOFTWARE DEPLOYMENT

The deployment strategy for the AI-Powered Quiz & Exam Platform was engineered to ensure a seamless, standalone installation of the desktop application while managing the complex dependencies associated with local machine learning models. As a Python-based application utilizing the PyQt5 framework for its graphical user interface, the deployment process required careful handling of both the Python environment and the external AI server. To eliminate the need for an external database server and simplify the deployment architecture, SQLite was selected as an embedded database. The `config.ini` file dynamically resolves the local database path, ensuring the `QuizPlatform.db` file is correctly instantiated on the target machine without requiring administrative database configuration.

The most critical component of the deployment was the configuration and initialization of the Ollama local LLM environment. The application specifically requires the mistral:7b-instruct-q4_0 model to function correctly. A `requirements.txt` file was meticulously maintained to govern the Python dependencies, dictating the installation of `PyQt5`, `requests`, and the `ollama` Python bindings. To streamline the initialization of the AI backend, a custom `start_ollama.bat` script was created. This script is responsible for launching the Ollama server with optimized environment variables tailored for the target machine, including `OLLAMA_NUM_THREADS` to maximize CPU utilization, `OLLAMA_VULKAN` to enable GPU acceleration, and `OLLAMA_MAX_LOADED_MODELS` to manage memory overhead efficiently.

To transition the application from a development script to an distributable executable, PyInstaller was utilized. This tool analyzes the `main.py` entry point and packages all necessary Python modules, UI assets, and custom exceptions into a standalone executable file, removing the requirement for the end-user to manually install a Python interpreter. The execution of the compiled application triggers an automated initialization sequence; upon its first run, the system automatically creates the required SQLite database tables via `database/db_setup.py` and injects essential seed data, such as the default Admin credentials, ensuring immediate operational readiness.

A comprehensive deployment checklist was strictly adhered to prior to final packaging. This checklist verified that all hardcoded paths were eliminated, debugging statements were purged, and the error logging system (`quiz_errors.log`) was configured to write to user-accessible directories. By combining an embedded database, a scripted AI server initialization, and a compiled PyInstaller executable, the software deployment process guarantees that the AI-Powered Quiz & Exam Platform can be reliably installed and executed on any compatible Windows machine, fulfilling the project's requirement for a robust desktop application.

# REFACTORING & LEGACY CODE REMOVAL

Throughout the development of the AI-Powered Quiz & Exam Platform, continuous refactoring and the systematic removal of legacy code were prioritized to maintain a robust, maintainable, and highly cohesive architecture. As the application's complexity grew, particularly with the integration of multiple AI features, early design implementations became bottlenecks. One of the most significant refactoring efforts involved the extraction of duplicate input validation logic. Initially, validation routines were scattered redundantly across various UI modules. To adhere to the DRY (Don't Repeat Yourself) principle, a dedicated `FormValidator` utility class was created in the `utils/` directory, centralizing all form validation and ensuring consistent user input handling throughout the application.

A major architectural overhaul was the dismantling of the monolithic 'God class' that initially handled all database operations. The legacy `DatabaseManager` was systematically split by implementing the Data Access Object (DAO) pattern. This resulted in the creation of specialized classes, including `StudentDAO`, `ExamDAO`, `QuestionDAO`, and `ResultDAO`, located within the `dao/` directory. This separation of concerns decoupled the database access logic, making the codebase significantly easier to test and maintain. Concurrently, the legacy flat-file approach used in early prototypes was completely removed and replaced with robust SQLite relational database queries, ensuring data integrity and transactional safety.

The integration of the AI features also underwent substantial refactoring. Early iterations featured direct `ask_ai()` network calls embedded directly within the UI event handlers, which caused unacceptable application freezing. These were completely refactored out of the UI layer. All AI communication was centralized into a single `ai_engine.py` module, and the direct calls were replaced with the `AIWorker` QThread pattern. This crucial refactoring decoupled the UI from the synchronous API requests, allowing the AI features to run asynchronously in background threads while communicating with the interface via PyQt signals. 

Finally, extensive code cleanup was conducted to ensure alignment with professional software engineering standards. All "magic numbers"—such as token limits and timeout durations—were excised from the operational code and replaced with named constants defined in `config.py` (e.g., `AI_NUM_CTX`, `AI_NUM_PREDICT`). Hardcoded database paths were similarly migrated to a configuration file (`config.ini`). A rigorous sweep was performed to remove dead code, including unused imports, deprecated placeholder functions, and diagnostic print statements. To finalize the codebase, comprehensive docstrings were added to all classes and functions, and PEP8 naming conventions were strictly enforced, resulting in a clean, professional, and highly readable project structure.

# UNIT TESTING

To guarantee the reliability and correctness of the AI-Powered Quiz & Exam Platform, a comprehensive suite of unit tests was engineered using the standard Python `unittest` framework. Given the complexity of the application, particularly its reliance on an external local LLM and a relational database, testing required a strategic approach to isolate components and verify their intrinsic logic. The test architecture was designed to validate the core business rules, database interactions, and the application's handling of edge cases without requiring the full system to be operational. This was achieved by creating dedicated test files within the `tests/` directory, including `test_auth.py`, `test_question.py`, `test_exam.py`, `test_grading.py`, and `test_ai_engine.py`.

A critical aspect of the testing strategy was the extensive use of the `unittest.mock` library, specifically for validating the AI components within `test_ai_engine.py`. Because the Ollama AI server is resource-intensive and its responses are inherently non-deterministic, running live AI models during unit tests was impractical. Instead, mock objects were deployed to simulate the AI API responses. This approach allowed the test suite to reliably verify the application's behavior under various conditions, such as simulating a successful JSON response (`test_ask_ai_success`) or forcing a simulated network failure (`test_ask_ai_connection_error`). Crucially, this mocking strategy ensures that all 30+ unit tests can execute and pass rapidly, even if the Ollama service is not currently running on the host machine.

The unit tests were meticulously designed to cover a wide spectrum of functional scenarios across the application's modules. In `test_auth.py`, standard authentication workflows were validated through tests such as `test_valid_login` and `test_invalid_login_raises_error`, ensuring secure access control. The `test_question.py` suite focused on data integrity, utilizing tests like `test_add_question_valid` and `test_question_empty_field_validation` to confirm that the `FormValidator` and DAOs functioned correctly. Furthermore, `test_exam.py` and `test_grading.py` validated the core educational mechanics, including `test_exam_creation` and `test_mcq_auto_grading`, which verified that the system accurately calculated student scores based on predefined rubrics.

Beyond functional correctness, the tests also verified critical system configurations and error handling protocols. Tests such as `test_keep_alive_in_payload` and `test_num_ctx_is_correct` were implemented to ensure that the AI engine consistently constructed the correct payload headers required for optimal LLM performance. The comprehensive nature of this unit testing regimen provided high confidence in the software's stability, ensuring that both the standard application logic and the complex, mocked AI interactions behaved exactly as designed prior to deployment.

# AUTOMATED TESTING

To maintain code quality and prevent the introduction of regressions during the iterative development of the AI-Powered Quiz & Exam Platform, a robust automated testing infrastructure was implemented. The foundation of this automation is the custom `run_tests.py` script, situated in the project's root directory. This script utilizes Python's `unittest.TestLoader` and `TestSuite` classes to automatically discover and execute all unit tests located within the `tests/` directory. By executing a single command (`python run_tests.py`), developers can instantly trigger the entire test suite, which culminates in a formatted console output detailing the test summary, including total tests run, passes, and specific failure tracebacks.

To elevate the automated testing from a local developer convenience to a systematic quality gate, a Continuous Integration (CI) pipeline was established using GitHub Actions. The configuration and rationale for this pipeline are thoroughly documented in the `CI_PIPELINE.md` file. The pipeline is configured to trigger automatically on every code push or pull request directed at the `develop` and `main` branches. This ensures that no new code can be integrated into the core application without first passing the rigorous automated verification process. The pipeline environment runs in an isolated container, guaranteeing that tests are evaluated in a clean, standardized setting free from local machine discrepancies.

The automated workflow defined within the CI pipeline executes a precise sequence of operational steps. It begins by checking out the repository code, setting up the required Python 3.10+ environment, and installing all necessary dependencies specified in the `requirements.txt` file. Once the environment is provisioned, the pipeline executes the `run_tests.py` script. A critical component of this automation is the strict enforcement of build integrity; if any single test fails—whether due to an authentication logic error or a mocked AI parsing failure—the entire CI build fails automatically, blocking the associated pull request from being merged. 

In addition to executing the tests, the automated pipeline utilizes the `coverage.py` tool to measure and report on the application's test coverage. This metric analyzes the percentage of the source code executed during the test suite run, highlighting areas of the codebase that lack sufficient testing. The generated coverage reports are uploaded as artifacts of the CI process, providing developers with actionable insights to improve the test suite. This comprehensive automated testing framework ensures that broken or insufficiently tested code is prevented from reaching production, significantly enhancing the overall reliability of the final software product.

# EXCEPTION HANDLING

A robust and comprehensive exception handling architecture was implemented throughout the AI-Powered Quiz & Exam Platform to ensure system stability and provide a graceful user experience during unexpected fault conditions. Recognizing that generic Python errors provide poor diagnostic value and confuse end-users, a suite of custom exception classes was engineered within the `exceptions.py` module. These custom classes—including `QuizAuthError`, `QuizValidationError`, `QuizDatabaseError`, `QuizAIError`, and `QuizExamError`—allow the application to raise semantically meaningful errors that precisely identify the subsystem where the failure occurred. This granular categorization significantly accelerates debugging and error resolution.

At the data persistence layer, rigorous exception handling is enforced to protect the integrity of the SQLite database. All operations within the DAO classes (Data Access Objects) are encapsulated within strict `try/except` blocks. If a database transaction fails due to a constraint violation or connection issue, the raw SQLite exception is caught, logged, and re-raised as a controlled `QuizDatabaseError`. This prevents database-level tracebacks from propagating up the call stack and ensures that the application can safely rollback failed transactions without crashing the main process.

The AI integration layer, encapsulated in `ai_engine.py`, features the most intricate exception handling due to its reliance on external API communications. All calls to the Ollama local LLM are designed to anticipate and cleanly handle specific network and parsing anomalies. A `ConnectionError` is caught and translated to inform the user that "Ollama is not running." Timeouts are captured and presented as "AI took too long to respond," while `JSONDecodeError` events—which occasionally occur when the LLM outputs malformed structures—trigger a graceful "Invalid format, try again" response. Crucially, the system is designed to degrade gracefully; if the AI engine experiences a critical failure, the core application remains fully functional, allowing students to continue taking standard exams without AI assistance.

To bridge the gap between technical error capture and user experience, a centralized logging and notification strategy was deployed. Every caught exception is recorded to a local `quiz_errors.log` file, complete with a precise timestamp, the relevant custom exception class, and the full stack trace for developer review. Conversely, the end-user is shielded entirely from raw Python tracebacks. Instead, when an error occurs, the `QThread` workers emit a custom `error_occurred` PyQt signal. The main UI thread intercepts this signal and displays a user-friendly error dialog box containing a clear, actionable message, ensuring that application instability is managed professionally and without crashing the graphical interface.

# PEER REVIEWS

Peer reviews were institutionalized as a critical quality assurance mechanism during the development of the AI-Powered Quiz & Exam Platform. Recognizing that complex integrations, particularly those involving asynchronous UI threading and local AI models, are prone to subtle logical errors, a formal peer review process was integrated into the Agile workflow. At the conclusion of each sprint, comprehensive code walkthroughs were conducted. These collaborative sessions allowed team members to explain their implementation logic, fostering a shared understanding of the evolving architecture and ensuring that the codebase adhered to the project's strict coding standards.

To enforce continuous quality control, formal Pull Request (PR) reviews were mandated on GitHub before any feature branch could be merged into the integration branch. These reviews were guided by a rigorous checklist designed to maintain consistency and reliability across the application. Reviewers explicitly checked for PEP8 compliance, code readability, the presence of appropriate custom exception handling, and the inclusion of corresponding unit tests for new features. Furthermore, the checklist strictly forbade the use of hardcoded values or "magic numbers," ensuring they were properly relocated to the `config.py` file, and verified that comprehensive docstrings were present on all new classes and functions.

The practical value of these peer reviews was repeatedly demonstrated by the identification and resolution of significant architectural flaws. For instance, a formal inspection conducted on the newly engineered `ai_engine.py` module revealed the absence of the crucial `keep_alive` parameter in the API payload, an omission that would have caused severe performance degradation due to model reloading. Similarly, a thorough code walkthrough of the `exam_taking_ui.py` module exposed a critical bug where synchronous AI calls were blocking the main event loop, causing UI freezing. This finding directly led to the implementation of the `QThread` pattern, drastically improving application responsiveness.

All review activities were meticulously documented to ensure accountability and track the evolution of code quality. A formal peer review log was maintained in the `PEER_REVIEW_LOG.md` file, recording the details of at least four major documented review sessions. This log detailed the specific sprint, the reviewer's identity, the module under scrutiny, the specific issues identified, and the subsequent technical resolution. By institutionalizing peer reviews—ranging from informal walkthroughs to strict PR gating—the project significantly minimized defects, improved architectural consistency, and ensured a higher caliber of software engineering throughout the development lifecycle.
